# Drive-Sync Threat Model — Red-Team Report

Scope: 33 adversarially-verified findings across the 6 Drive-synced apps. 1 refuted, 32 confirmed as real or partial. No finding produced an irreversible whole-Drive wipe — every "catastrophic" claim was downgraded by the recovery layers. But there are multiple **real, silent, partial-data-loss** paths today, and several more that are **one plausible agentic edit** from biting.

---

## 1. Resilience verdict

### (a) "Losing data / a bug wiping the user's Drive" → **split verdict**

- **Whole-Drive destruction (a bug nuking files outside the app folder): Strong.** No confirmed path irreversibly destroys user data at scale. Three independent floors hold: the `drive.file` OAuth scope (Google refuses `trashed:true` on any file the app didn't create — this alone refuted the Gesture foreign-folder trash, #19), trash-only deletes (never `files.delete`, ~30-day recovery), and the ancestry/stewardship filters confining every trash sweep to app-owned trees.
- **Silent partial data loss: Adequate, trending Needs-work under agentic development.** This is where the real findings cluster. Confirmed _today_: whole-row last-writer-wins silently drops non-exercise song fields and second videos on performances (#1, #2); the history-recovery UI scans the **oldest** 40 revisions and structurally cannot see its own keepForever pins (#25); a single user-trashed sidecar aborts hydration of an entire library and pauses sync (#12); the manual "Back up to Drive" button clobbers cloud when a pull hit an unresolved conflict (#9); Gesture "Refresh" pushes without a preceding pull (#10). None is a wipe, but each loses _some_ real user content, and the misleading conflict/confirm copy means users won't invoke recovery.

**Net (a): Adequate.** Catastrophe is well-defended; silent erosion is not, and the merge/recovery layers have real holes.

### (b) "Near-impossible for normal usage to get the user banned" → **Adequate**

Current idle/steady-state volume (~60–600 req/hr across all apps) is far under Drive's per-user quota, so _today_ ban risk is low. But there is **no absolute request-rate governor anywhere** (#28, #29, #30, #32): retry paths ignore Google's `Retry-After` header and add no jitter (synchronized retry waves — the exact abuse-detection signature), and every silent auto-pull unconditionally re-pushes `progress.json` (#31). Safety rests entirely on the _values_ of a few editable timer constants with no test floor and no runtime ceiling. A one-line "snappier sync" or "parallelize shard upload" edit scales linearly into abuse territory with no backstop. Given the stated goal ("even with constant agentic development"), the absence of a structural ceiling is the gap.

### The honest OAuth limit

The "can't wipe the whole Drive" guarantee is **borrowed from Google's server-side `drive.file` scope enforcement, not from app code.** Findings #19 (Picker-based folder linking would grant write to foreign subtrees) and #20 (widening dedup's ancestor set to override folders) show that a single scope/feature change moves the floor. The app-level stewardship checks are already holed for nested/absorbed folders (#19) — they pass today only because Google rejects the trash. **Any future move to `drive.file`-via-Picker on user-chosen folders, or to a broader scope, must be treated as removing a load-bearing defense** and gated accordingly.

## 3. Refuted attacks (defenses working)

Evidence of real strength — these were probed and the existing layers hold:

- **Corrupt/wrong-schema tombstone store → resurrection (#24, REFUTED).** `prepareGestureDriveMerge` re-seeds local tombstones from the **remote envelope** before every merge, and the merge filter is the local union, not the pushed set. The cloud envelope is an independently-schema-versioned durable source of truth; a corrupt local cache repopulates on the next pull, and `allowAutoPush` prevents a push preceding that pull. Round-trip tests already catch an inconsistent read/write schema bump. Stopped by **Layer 4 + Layer 2**.
- **Whole-Drive foreign-file trash via Gesture link+reconcile (#19).** Stopped server-side by the **`drive.file` OAuth scope** — `driveTrashFile` gets 403 on any non-app-created file. (App-level stewardship is _holed_ for the nested case; the scope is what saves it — hence hardening still recommended.)
- **Truncated-upload → healthy envelope ref (#14 exploit path).** Stopped by the **resumable protocol itself** — Drive only returns a finalized id after receiving all bytes; a gapped upload yields 308-not-completed and throws. (App-level parity gap remains for a future hand-built multipart path.)
- **Concurrent-push mid-trash publishing dangling refs (#13 failure-mode 2).** Stopped by `labsBlockingJobsActive()` — the delete dialog starts a non-silent blocking job that gates auto-push for the whole operation.
- **Empty-overwrite / accidental clobber → permanent loss.** Repeatedly stopped by **Layer 6**: daily keepForever pin on `progress.json` + Drive revision history + in-app history recovery downgraded every "catastrophic" merge/tombstone finding to recoverable.

## 4. Top hardening recommendations (highest leverage, whole-class)

Ranked by how many future agentic bugs each prevents:

1. **Compile-time merge-policy exhaustiveness per synced entity** (kills the entire #3/#1/#2 class). `satisfies Record<keyof EncoreSong, MergePolicy>` (and the same for `EncorePerformance`, `LyreflyProject`) so **no new synced field can compile without a conscious merge disposition**, and the recovery-side union shares the map. Converts the prose "any new field → add to merge" checklist into a gate an agent cannot skip. This is the single highest-leverage change — it defends the most-likely future-feature bug across the repo.

2. **Move the auto-push gate _inside_ `flushDriveWrite`** (kills #9/#10/#11/#18 as a class). Make the primitive fail-closed: refuse a write when `!allowAutoPush()` unless an explicit `intentionalReplace` token is threaded from the one legitimate replace path, and switch the If-Match precondition from _current_ cloud etag to _last-pulled_ etag. Every present and future direct caller then inherits "no push before a reconciling pull" structurally instead of by convention. Back with a choke-point test asserting no write on a never-pulled session.

3. **One shared, Retry-After-honoring, bounded-concurrency Drive request wrapper at `driveFetch`** (kills the entire #28/#29/#30/#31/#32 ban class). A module-level throttle + token bucket + jittered backoff that _all_ reads and writes funnel through, so one 429 pauses all Drive traffic and no timer-constant or fan-out edit can scale past a hard ceiling. Add a `known-gaps` row: "API-abuse/rate-limiting is not covered by the 10 data-loss layers."

4. **Guard-parity contract test in `labsPortfolioDriveHookGuardrails.test.ts`** asserting, for **every app**, that `downloadSidecars` tolerates a single 404 blob without aborting the batch or pausing sync (#12), and that push has a filled>empty check + size verification (#14/#26). Parity divergence between apps is a recurring root cause; make it a test, not a code-review hope.

5. **Fix the history-recovery scan ordering + assert pinner/scanner agree** (#25). Sort newest-first before the 40-cap and pin the invariant with a test that a recently-lost + keepForever-pinned entity is surfaced. This restores the _entire last-resort recovery net_, which is currently blind to its own pins.

6. **Decouple placeholder detection from `schemaVersion` and enforce `schemaVersion>=1` across all `buildEnvelope`s** (#23). One predicate change + one enumeration test makes a whole class of "new app / migration clobbers cloud" bugs impossible.

7. **Ancestry/provenance assertions pinned by argument, not return value** (#19/#20/#21). Extract a single `trashableRootIds` helper shared by delete + dedup, add per-descendant provenance checks independent of OAuth scope, and pin the ancestor-set _argument_ in tests so a widening edit fails CI. This de-risks the OAuth "honest limit" — it stops relying on Google's scope as the sole floor against foreign-file trash.

**Cross-cutting note:** the recurring failure signature across nearly every real finding is **misleading UX copy + silent loss** (conflict dialogs naming the one safe field, confirm dialogs promising local-only wipes, success toasts after a clobber). Even where recovery layers technically save the bytes, users never invoke recovery because nothing told them anything was lost. A pass to make every destructive/lossy path _honest in its copy and loud on divergence_ would raise the practical floor as much as any single test.
